# Phase 1.2: Vendor Dashboard Initial CRUD

## Overview
This phase builds the operational core for vendors. We will implement a data-dense dashboard within the Next.js app, allowing vendors to manage their inventory (CRUD operations) and view basic business metrics.

## Scope
- **Vendor Layout**: Implementation of a persistent sidebar and dashboard header in `apps/web/src/app/(vendor)`.
- **Inventory Management**:
  - `GET /api/v1/products/vendor/{id}`: Fetch vendor-specific products.
  - `POST /api/v1/products`: Create new products.
  - `PATCH /api/v1/products/{id}`: Update existing products.
  - `DELETE /api/v1/products/{id}`: Archive/Remove products.
- **Product Forms**: Complex forms with validation (Zod) and shadcn UI components (Input, Textarea, Select).

## Technical Decisions
- **Framework**: Next.js 16 (App Router) using the `(vendor)` route group.
- **Components**: shadcn/ui (Table, Form, Dialog/Modal for CRUD).
- **Forms**: React Hook Form + Zod for validation.
- **Data Fetching**: TanStack Query mutations (`useMutation`) for CRUD actions.
- **Real-time Feedback**: Sonner (Toast) notifications for success/error states.

## Constraints
- **Multi-vendor Isolation**: Ensure vendors can only see and edit their own products (handled by backend, but UI must reflect this).
- **Aesthetics**: Maintain the premium, minimalist design established in the storefront.

## Success Criteria
- [ ] Vendor can view a list of their products in a responsive table.
- [ ] Vendor can add a new product with full validation.
- [ ] Vendor can edit an existing product.
- [ ] Vendor can delete a product with a confirmation prompt.
