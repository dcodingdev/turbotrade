# Phase 2.1 Discussion Log: Order Management & Timelines

## Design Decisions

### D-01: Per-Item Status Tracking
- **Decision**: Add a `status` field to each `OrderItem` within the `Order` model.
- **Rationale**: Enables multi-vendor fulfillment where different vendors can ship their parts of an order at different times without waiting for others.
- **Statuses**: `PENDING`, `PROCESSING`, `SHIPPED`, `DELIVERED`, `CANCELLED`.

### D-02: Roadmap Alignment
- **Decision**: Follow the roadmap states: `Processing` -> `Shipped` -> `Delivered`.
- **Implementation**: The backend will support these transitions.

### D-03: Vendor Isolation
- **Decision**: The `GET /api/v1/orders/vendor` endpoint will return the full order but the frontend will filter/highlight items owned by the requesting vendor.
- **Security**: Vendors can only transition the status of items where `item.vendor === req.user._id`.

### D-04: Global State Derivation
- **Decision**: The top-level `orderStatus` will reflect the overall lifecycle (e.g., `PAID`, `COMPLETED`, `CANCELLED`) while item-level statuses drive the granular tracking.
