# Phase 2.2: Real-time Chat - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers a full-stack real-time communication system between customers and vendors. It includes the WebSocket server implementation in the `chat-service`, the frontend client integration in the `apps/web` (Next.js), interactive chat windows, and an offline notification system.

</domain>

<decisions>
## Implementation Decisions

### Protocol & Infrastructure
- **D-01:** Use **Socket.io** for real-time communication. This choice provides robust reconnection logic, room management for private chats, and scalability via a Redis adapter.
- **D-02:** Integrate `@socket.io/redis-adapter` in the `chat-service` to allow the WebSocket server to scale across multiple microservice instances.

### UI & UX Layout
- **D-03:** **Customer Interface:** Implement a floating global chat widget (bubble) accessible from any page, plus an embedded chat view on Order Tracking pages.
- **D-04:** **Vendor Interface:** Create a dedicated `/dashboard/chat` full-screen view within the Vendor Dashboard to manage high volumes of customer inquiries.
- **D-05:** Use **shadcn/ui** and **Tailwind CSS v4** for all chat components to maintain the premium, minimalist design of the platform.

### Authentication & Security
- **D-06:** Implement **Handshake JWT Authentication**. The `chat-service` will validate the `Authorization` header or query token during the initial connection handshake using the shared `JWT_ACCESS_SECRET`.

### Notification Strategy
- **D-07:** Use **Sonner (Toasts)** for real-time in-app notifications when a user is active.
- **D-08:** Implement an **Email Fallback**. If a message remains unread for more than 5 minutes (monitored via a background worker or Redis TTL), the `email-service` will be triggered to notify the recipient.

</decisions>

## Existing Code Insights

### Reusable Assets
- `packages/database`: Provides Mongoose connection logic used by the chat service.
- `packages/common`: Contains auth middleware and utility functions for JWT.
- `apps/web/src/components/ui`: shadcn/ui components for building the chat interface.

### Established Patterns
- **Microservices**: All services use Express with a shared `@repo/common` library.
- **Mongoose Models**: UUID-based `_id` fields are standardized across the `chat-service`.

- `chat-service/src/server.ts`: Entry point for adding the Socket.io server.
