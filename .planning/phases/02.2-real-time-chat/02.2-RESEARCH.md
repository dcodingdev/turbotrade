# Phase 2.2: Real-time Chat - Research

## Overview
This research defines the technical implementation for the real-time communication layer between customers and vendors. It focuses on using Socket.io with a Redis adapter for scalability, handshake-based JWT authentication, and a resilient Next.js client integration.

## Technical Implementation Guide

### 1. WebSocket Infrastructure (Socket.io + Redis)
To ensure horizontal scalability across multiple microservice instances, the `chat-service` will use the Redis adapter. This allows events emitted on one instance to be broadcast to clients connected to other instances.

- **Adapter**: `@socket.io/redis-adapter`
- **Client**: `ioredis` (Preferred for its robust feature set and cluster support)
- **Pattern**:
    ```typescript
    import { Server } from "socket.io";
    import { createAdapter } from "@socket.io/redis-adapter";
    import Redis from "ioredis";

    const pubClient = new Redis(process.env.REDIS_URL);
    const subClient = pubClient.duplicate(); // subClient must be a duplicate for subscription mode

    const io = new Server(httpServer, {
      adapter: createAdapter(pubClient, subClient),
      cors: { origin: process.env.FRONTEND_URL, credentials: true }
    });
    ```

### 2. Handshake Authentication
Authentication will be enforced at the initial connection (handshake) level.

- **Mechanism**: The client will pass the JWT in the `auth` object during connection.
- **Validation**:
    ```typescript
    io.use((socket, next) => {
      const token = socket.handshake.auth.token || socket.handshake.headers['authorization'];
      if (!token) return next(new Error("Authentication error: No token provided"));

      try {
        const decoded = jwt.verify(token.replace("Bearer ", ""), process.env.JWT_ACCESS_SECRET);
        socket.data.user = decoded; // Attach user info to the socket context
        next();
      } catch (err) {
        next(new Error("Authentication error: Invalid token"));
      }
    });
    ```

### 3. Next.js Client Hook (`useChat`)
A custom hook in `apps/web` will manage the connection lifecycle and room synchronization.

- **Responsibilities**:
    - Manage socket connection/disconnection.
    - Join the appropriate conversation room on mount.
    - Synchronize real-time messages with the TanStack Query cache.
    - Handle reconnect events.
- **Pattern**:
    ```typescript
    export const useChat = (conversationId: string) => {
      const [messages, setMessages] = useState([]);
      const socketRef = useRef<Socket | null>(null);

      useEffect(() => {
        const socket = io(process.env.NEXT_PUBLIC_CHAT_URL, {
          auth: { token: getAccessToken() }
        });

        socket.emit("join_room", { conversationId });

        socket.on("new_message", (message) => {
          // Sync with TanStack Query or local state
          queryClient.setQueryData(["messages", conversationId], (old) => [...old, message]);
        });

        socketRef.current = socket;
        return () => { socket.disconnect(); };
      }, [conversationId]);
    };
    ```

### 4. Interactive Features
- **Typing Indicators**: Throttled `typing` events emitted by the client, broadcast to the room using `socket.to(room).emit("user_typing", ...)`.
- **Read Receipts**: When a message enters the viewport (via IntersectionObserver), emit a `message_read` event. The server updates the DB and broadcasts to the room.

### 5. Offline Notifications (Email Fallback)
To ensure messages aren't missed when a user is away:
1. **Detection**: The server checks if the recipient is currently connected to the conversation room.
2. **Persistence**: Use a delayed task (e.g. BullMQ or a simple Redis-based delay) to wait 5 minutes.
3. **Verification**: If the message status is still `unread` after 5 minutes, trigger an event to the `email-service`.
4. **Cancellation**: If a `message_read` event is received before the timeout, cancel the pending notification task.
