# Phase 1.3: Validation Strategy

## Dimensions of Verification

### 1. Requirements Completeness
- [ ] Cart state persists across page reloads (Zustand + localStorage).
- [ ] Items in the cart are grouped by vendor.
- [ ] Multi-step checkout wizard allows progression from Cart -> Shipping -> Payment -> Success.
- [ ] Integration with Stripe `PaymentElement` successfully processes a mock payment.
- [ ] Real-time stock locking gracefully handles failure (displays modal, redirects to cart).

### 2. Architectural Adherence
- [ ] Next.js App Router conventions are followed for checkout routes.
- [ ] No direct database calls from the frontend; relies entirely on `order-service` and `payment-service` REST endpoints.
- [ ] Styling strictly uses Tailwind CSS v4 and the existing globals.css OKLCH tokens.

### 3. Edge Case Handling
- [ ] Out-of-stock items encountered during `POST /api/v1/orders` properly display the resolution modal.
- [ ] Declined cards in Stripe are caught and the user is prompted to try another payment method.
- [ ] Cart is correctly cleared upon successful payment.
- [ ] Empty cart state is handled gracefully, preventing checkout initiation.

## Nyquist Criteria
**Dimension 8:** The resulting PLAN.md MUST include explicit verification steps for the 400 stock error flow, ensuring that the `createOrder` endpoint's exact failure response triggers the inline modal.
