# Phase 1.3: Unified Cart & Checkout Flow - Research

**Objective:** Understand the backend requirements and necessary frontend architecture to implement the unified cart and checkout flow.

## 1. Backend API Review

### Order Service (`createOrder`)
- **Endpoint:** `POST /api/v1/orders`
- **Payload Requirements:**
  - `items`: Array of objects containing `product` (ID), `vendor` (ID), `quantity`, and `priceAtPurchase`.
- **Behavior:**
  - Calculates the total amount.
  - Creates an order in `PENDING` state.
  - Sequentially reserves stock for each item via the Product Service (`POST /api/stock/:productId/reserve`).
  - **Failure Handling:** If any stock reservation fails, it rolls back previously reserved items, updates the order status to `CANCELLED`, and returns a `400 Bad Request` with `{ success: false, message: "Stock reservation failed. Order cancelled.", error: "..." }`.

### Payment Service (`createPaymentIntent` & Webhooks)
- **Endpoint:** `POST /api/v1/payments`
- **Payload Requirements:**
  - `orderId`: The ID of the order created in the previous step.
  - `amount`: Total amount to be charged.
  - `gateway`: String, typically `"STRIPE"`.
  - `currency`: Defaults to `"USD"`.
- **Behavior:**
  - Calls Stripe's `paymentIntents.create`.
  - Creates a `Payment` record in MongoDB in `PENDING` state.
  - Returns `{ success: true, gatewayData: { id, client_secret, ... }, paymentId }`. The `client_secret` from `gatewayData` is required for the Stripe Elements UI.
- **Webhook Integration:**
  - Stripe Webhook endpoint is active (`POST /api/v1/payments/webhook/stripe`).
  - Upon `payment_intent.succeeded`, it finalizes the payment in MongoDB and publishes a `payment.success` event via RabbitMQ to trigger stock confirmation and order updates.

## 2. Frontend Architecture

### Cart State Management (Zustand)
- We need a robust client-side store using Zustand.
- The state should include:
  - `items`: The list of products added to the cart.
  - `addItem`, `removeItem`, `updateQuantity`, `clearCart`.
- **Persistence:** Use Zustand's `persist` middleware to save the cart state in `localStorage` (`cart-storage`).
- **Grouping by Vendor:** A selector or getter in the store to return items grouped by `vendorId` to fulfill the D-01 decision.

### Multi-Step Checkout Wizard
- **Flow:**
  1. **Cart Review:** Review grouped items and adjust quantities.
  2. **Shipping/Details:** Enter shipping address and details.
  3. **Payment:** Calls `/api/v1/orders` to lock stock. If successful, calls `/api/v1/payments` to get the Stripe `client_secret`, then mounts the Stripe `PaymentElement`.
  4. **Review/Confirmation:** Display success message and clear cart.

### Out-of-Stock / 409 Handling
- The backend currently returns a `400 Bad Request` when stock reservation fails (not a `409 Conflict`).
- The frontend must catch this specific error (`"Stock reservation failed. Order cancelled."`) during the Payment step initialization.
- Upon catching it, an inline modal should inform the user which items are unavailable.
- After dismissal, the user should be redirected back to the Cart Review step.

### Stripe Elements Integration
- Use `@stripe/stripe-js` and `@stripe/react-stripe-js`.
- Wrap the payment step in `<Elements stripe={stripePromise} options={{ clientSecret }}>`.
- Mount the `<PaymentElement />` and handle submission via `stripe.confirmPayment`.

## 3. Recommended Plan Structure
1. Set up Zustand cart store with persistence.
2. Build the Cart Review page with items grouped by vendor.
3. Build the multi-step checkout wizard structure.
4. Implement the API integration for `createOrder` with stock failure handling (modal + redirect).
5. Integrate Stripe Payment Element in the final checkout step.
